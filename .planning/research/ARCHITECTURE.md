# Architecture Patterns

**Domain:** Offline-first PWA with relational local data (speech development tracker)
**Project:** Little Words
**Researched:** 2026-06-30
**Confidence:** MEDIUM (Dexie/React Router/vite-plugin-pwa official docs); LOW (build-order rationale, folder structure)

---

## Recommended Architecture

Three horizontal layers, each with a hard one-way dependency rule:

```
┌─────────────────────────────────────────────────┐
│  UI Layer                                        │
│  src/pages/ (route shells)                       │
│  src/features/<name>/{components,hooks}          │
│  src/shared/components/                          │
│  communicates with ↓ only                        │
├─────────────────────────────────────────────────┤
│  Service Layer                                   │
│  src/db/services/<entity>.service.ts            │
│  Thin async functions wrapping Dexie calls       │
│  communicates with ↓ only                        │
├─────────────────────────────────────────────────┤
│  Data Layer                                      │
│  src/db/db.ts           (Dexie singleton)        │
│  src/db/schema.ts       (schema version def)     │
│  src/db/types.ts        (entity interfaces)      │
│  IndexedDB via Dexie v4                          │
└─────────────────────────────────────────────────┘

Cross-cutting:
  src/i18n/           (initialized before React mounts)
  src/router/         (createHashRouter definition)
  public/sw.js        (generated by vite-plugin-pwa)
```

## Component Boundaries

### What talks to what

| Component | Reads from | Writes to | Notes |
|-----------|-----------|-----------|-------|
| Page components (`src/pages/`) | Feature hooks | — | Route shells; own no business logic |
| Feature components (`src/features/<name>/components/`) | Feature hooks, shared UI | Service functions via callbacks | Never import db directly |
| Feature hooks (`src/features/<name>/hooks/`) | `useLiveQuery` + service fns | — | Reactive queries live here |
| Service functions (`src/db/services/`) | `db` singleton | `db` singleton | Pure async; no React |
| `db.ts` | — | IndexedDB | Dexie subclass singleton |
| Router (`src/router/index.ts`) | Page components | — | `createHashRouter`, no data loaders needed for MVP |
| i18n (`src/i18n/index.ts`) | `src/locales/` JSON | — | Initialized before `ReactDOM.render` |
| Service Worker (generated) | Vite build output | Cache Storage | Managed by vite-plugin-pwa; not hand-written |

### Data Flow Direction

```
User action
  → Feature component event handler
  → Service function (db.table.add / put / delete)
  → IndexedDB mutation
  → useLiveQuery observes change
  → React re-renders affected components
```

Reads never go through services — `useLiveQuery` queries `db` directly inside feature hooks (or inline if the query is trivial). This is the Dexie-idiomatic pattern: services handle writes; hooks handle reads.

## IndexedDB Schema Design

### Entity Model

```
ChildProfile      (1)  ─────────────────────  (1) app-wide singleton
WordForm          (N)
Meaning           (N)  independent entity; survives word form deletion
WordFormMeaning   junction: WordForm ↔ Meaning (M:N)
MeaningCategory   junction: Meaning ↔ Category (M:N)
Gesture           (N)  separate entity, not linked to WordForm/Meaning
```

### Dexie Schema Definition

```typescript
// src/db/db.ts
import Dexie, { type EntityTable } from 'dexie';
import type {
  ChildProfile, WordForm, Meaning,
  WordFormMeaning, Gesture
} from './types';

export class AppDB extends Dexie {
  childProfile!: EntityTable<ChildProfile, 'id'>;
  wordForms!:    EntityTable<WordForm,     'id'>;
  meanings!:     EntityTable<Meaning,      'id'>;
  wordFormMeanings!: EntityTable<WordFormMeaning, 'id'>;
  gestures!:     EntityTable<Gesture,      'id'>;

  constructor() {
    super('LittleWordsDB');
    this.version(1).stores({
      // Primary key first, then indexed fields
      childProfile:     '++id',
      wordForms:        '++id, form, createdAt',
      meanings:         '++id, text, isActive, firstUseDate, lastUseDate',
      // Junction table: compound index enables bidirectional lookups
      wordFormMeanings: '++id, wordFormId, meaningId, [wordFormId+meaningId]',
      gestures:         '++id, description, firstObservedDate',
    });
  }
}

export const db = new AppDB();
```

### Types

```typescript
// src/db/types.ts
export interface ChildProfile {
  id?: number;
  name: string;
  birthDate: string;       // ISO date string
  languages: string[];
  parentNotes: string;
  createdAt: string;
}

export interface WordForm {
  id?: number;
  form: string;            // e.g. "pa"
  createdAt: string;
}

export interface Meaning {
  id?: number;
  text: string;            // e.g. "goodbye"
  categories: string[];    // fixed category keys, e.g. ["verbs"]
  firstUseType: 'spontaneous' | 'repeated';
  isActive: boolean;
  firstUseDate: string;
  lastUseDate: string;
}

export interface WordFormMeaning {
  id?: number;
  wordFormId: number;
  meaningId: number;
}

export interface Gesture {
  id?: number;
  description: string;
  firstObservedDate: string;
  lastObservedDate: string;
}
```

### Junction Table Query Patterns

```typescript
// All meanings for a word form (used on WordForm detail page)
const meanings = await db.wordFormMeanings
  .where('wordFormId').equals(wordFormId)
  .toArray()
  .then(links => db.meanings.bulkGet(links.map(l => l.meaningId)));

// All word forms that express a meaning (used on Meaning detail page)
const wordForms = await db.wordFormMeanings
  .where('meaningId').equals(meaningId)
  .toArray()
  .then(links => db.wordForms.bulkGet(links.map(l => l.wordFormId)));

// Deleting a word form: remove junction rows, leave meanings intact
await db.transaction('rw', [db.wordForms, db.wordFormMeanings], async () => {
  await db.wordFormMeanings.where('wordFormId').equals(wordFormId).delete();
  await db.wordForms.delete(wordFormId);
});
```

### Version Migration Pattern

```typescript
this.version(1).stores({ /* initial schema */ });

this.version(2).stores({
  // Add new table or index — only additions; never remove an index
  // from a prior version block (it breaks downgrade paths)
  meanings: '++id, text, isActive, firstUseDate, lastUseDate, *categories',
}).upgrade(tx => {
  // populate new fields on existing records if needed
  return tx.table('meanings').toCollection().modify(m => {
    if (!m.categories) m.categories = [];
  });
});
```

## Data Layer Architecture

### Pattern: Singleton DB + Thin Services + Feature Hooks

Reject three extremes:
- **Direct Dexie calls in components** — couples UI to DB; makes testing impossible.
- **Full repository pattern with interfaces** — over-engineered for a single-device offline app with one data source.
- **Dexie hooks (db.on('creating'))** — useful for computed fields but adds hidden side-effects; avoid for business logic.

Use the middle path:

```
src/db/
  db.ts           ← Dexie subclass + singleton export
  types.ts        ← Entity interfaces
  services/
    wordForm.service.ts   ← addWordForm(), deleteWordForm(), linkMeaning()
    meaning.service.ts    ← addMeaning(), toggleActive(), updateLastUse()
    gesture.service.ts    ← addGesture(), updateGesture()
    profile.service.ts    ← saveProfile(), getProfile()
    export.service.ts     ← toJSON(), toCSV()
    import.service.ts     ← fromJSON()

src/features/<name>/hooks/
  useWordForms.ts       ← useLiveQuery(() => db.wordForms.toArray(), [])
  useMeanings.ts        ← useLiveQuery(() => db.meanings.where(...).toArray(), [deps])
  useDashboardStats.ts  ← useLiveQuery for multiple aggregates
```

Service functions are plain async functions — they can be unit tested without React. Feature hooks wrap `useLiveQuery` and expose the data to components. Components call service functions on user actions.

### Page-to-Feature Wiring

Each page is a route shell. It:
1. Renders a layout frame (header/bottom nav)
2. Composes one or more feature components
3. Passes no props down — feature hooks pull from db directly

```tsx
// src/pages/DashboardPage.tsx
export function DashboardPage() {
  return (
    <AppLayout>
      <DashboardStats />    {/* feature component */}
      <RecentlyAdded />
      <ReviewSuggestions />
    </AppLayout>
  );
}
```

## Service Worker Strategy

### Strategy: generateSW with full precache

For a fully offline SPA with no external API calls, the correct strategy is:

- **Mode:** `generateSW` (Workbox generates the SW; no hand-written sw.ts needed)
- **Registration:** `registerType: 'autoUpdate'` — new SW activates immediately on next navigation
- **Precache everything:** `workbox.globPatterns: ['**/*']` and `includeAssets: ['**/*']`
- **Navigation fallback:** `navigateFallback: 'index.html'` — all hash routes return the app shell
- **No runtime caching rules needed** — all assets are bundled; there is no network API to cache

```typescript
// vite.config.ts (VitePWA section)
VitePWA({
  registerType: 'autoUpdate',
  includeAssets: ['**/*'],
  workbox: {
    globPatterns: ['**/*'],
    navigateFallback: 'index.html',
  },
  manifest: {
    name: 'Little Words',
    short_name: 'Little Words',
    description: 'Track your child\'s speech development',
    theme_color: '#ffffff',
    background_color: '#ffffff',
    display: 'standalone',
    scope: '/little-words/',
    start_url: '/little-words/',
    icons: [
      // IMPORTANT: prefix icon src with base path — not auto-prepended
      { src: '/little-words/icons/pwa-192x192.png', sizes: '192x192', type: 'image/png' },
      { src: '/little-words/icons/pwa-512x512.png', sizes: '512x512', type: 'image/png' },
    ],
  },
})
```

### Why not stale-while-revalidate for the shell?

Stale-while-revalidate makes sense when the app shell can be updated in the background while serving the cached version. For a fully offline-first app with no server dependency, it adds complexity without benefit. The `autoUpdate` registration strategy handles updates cleanly: when the user is online and a new build deploys, the SW updates on the next page load.

## GitHub Pages Deployment Architecture

### Vite Configuration

```typescript
// vite.config.ts
export default defineConfig({
  base: '/little-words/',   // must match GitHub repo name
  build: {
    outDir: 'dist',
  },
  // ... plugins
})
```

### Known Pitfalls with Base Path + PWA

1. **Icon paths in manifest are not auto-prefixed** — write full paths including base in every icon `src`.
2. **sw.js scope** — explicitly set `scope: '/little-words/'` in the manifest; do not rely on defaults.
3. **Hash routing + base** — with `createHashRouter`, set no `basename` option; the hash routing means all navigation uses `/#/path` and the base is only relevant for the HTML file URL.
4. **gh-pages branch deployment** — Vite outputs to `dist/`; GitHub Actions workflow should push `dist/` to the `gh-pages` branch. Use `actions/deploy-pages` or `peaceiris/actions-gh-pages`.

### Build Output Structure

```
dist/
  index.html              ← app shell; loaded for any /#/* URL
  assets/
    index-[hash].js
    index-[hash].css
  sw.js                   ← generated service worker
  manifest.webmanifest    ← PWA manifest
  icons/
    pwa-192x192.png
    pwa-512x512.png
```

## Hash Routing Setup

### Router Definition

```typescript
// src/router/index.ts
import { createHashRouter } from 'react-router';
import { DashboardPage } from '../pages/DashboardPage';
import { OnboardingPage } from '../pages/OnboardingPage';
// ... other pages

export const router = createHashRouter([
  {
    path: '/',
    element: <RootLayout />,     // bottom nav + outlet
    children: [
      { index: true, element: <Navigate to="/dashboard" replace /> },
      { path: 'dashboard', element: <DashboardPage /> },
      { path: 'meanings', element: <MeaningsPage /> },
      { path: 'meanings/:id', element: <MeaningDetailPage /> },
      { path: 'word-forms', element: <WordFormsPage /> },
      { path: 'word-forms/:id', element: <WordFormDetailPage /> },
      { path: 'categories', element: <CategoriesPage /> },
      { path: 'timeline', element: <TimelinePage /> },
      { path: 'gestures', element: <GesturesPage /> },
      { path: 'settings', element: <SettingsPage /> },
    ],
  },
  { path: '/onboarding', element: <OnboardingPage /> },
]);
```

`createHashRouter` is preferred over the legacy `<HashRouter>` component because it supports the data router API (future loaders, actions) and lazy routes via `lazy()`.

### Onboarding Gate

```tsx
// src/app/App.tsx
function App() {
  const profile = useLiveQuery(() => db.childProfile.toArray());
  if (profile === undefined) return <Spinner />;   // loading
  if (profile.length === 0) return <OnboardingPage />;
  return <RouterProvider router={router} />;
}
```

Onboarding is rendered outside the main router to avoid hash-route complexity and ensure it runs before any route resolution.

## i18n Architecture

### Decision: Bundled resources, namespaced

For two languages and ~200 translation keys total, HTTP-lazy-loading translation files adds roundtrips and complexity with no meaningful bundle-size benefit. Bundle both locale files in the main chunk.

### Namespace Structure

```
src/i18n/
  index.ts              ← i18next init + language detection
  locales/
    en/
      common.json       ← navigation labels, shared UI strings, category names
      onboarding.json   ← wizard steps, field labels
      report.json       ← doctor report template strings
    pl/
      common.json
      onboarding.json
      report.json
```

Three namespaces because they have distinct lifecycles: `common` is loaded everywhere, `onboarding` only during setup, `report` only when generating a report. This grouping also makes handoff to a human translator straightforward.

### Init Pattern

```typescript
// src/i18n/index.ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import enCommon from './locales/en/common.json';
import enOnboarding from './locales/en/onboarding.json';
import enReport from './locales/en/report.json';
import plCommon from './locales/pl/common.json';
import plOnboarding from './locales/pl/onboarding.json';
import plReport from './locales/pl/report.json';

i18n.use(initReactI18next).init({
  resources: {
    en: { common: enCommon, onboarding: enOnboarding, report: enReport },
    pl: { common: plCommon, onboarding: plOnboarding, report: plReport },
  },
  lng: localStorage.getItem('language') ?? 'pl',  // Polish default
  fallbackLng: 'en',
  defaultNS: 'common',
  interpolation: { escapeValue: false },
});

export default i18n;
```

Import this file in `src/main.tsx` before `ReactDOM.createRoot`. Language changes write to `localStorage` and call `i18n.changeLanguage()`.

## Suggested Build Order

Dependencies flow downward. Each item can only be started when its dependencies are complete.

| Order | Layer / Feature | Depends on | Rationale |
|-------|----------------|-----------|-----------|
| 1 | DB types + schema (`src/db/types.ts`, `db.ts`) | — | Everything else imports from here |
| 2 | Service functions (`src/db/services/`) | DB schema | Needed by hooks; no UI yet |
| 3 | i18n init + locale files | — | Parallel with DB; no dependencies |
| 4 | Router + layout shell (`RootLayout`, `BottomNav`) | i18n, router | Proves routing works before feature work begins |
| 5 | Shared UI primitives (`FAB`, `BottomSheet`, `DateField`) | i18n | Used by multiple features; extract early to avoid duplication |
| 6 | Onboarding feature | DB services, shared UI, i18n | Dashboard reads profile; must exist first |
| 7 | Dashboard feature | DB services (reads meanings/word forms), onboarding (profile) | Primary screen; validates data model with real data |
| 8 | Add-entry flow (FAB → BottomSheet → word form + meaning) | Shared UI, DB services, Dashboard | Core data creation loop |
| 9 | Meanings list + detail + active toggle | DB services | Browsing the primary entity |
| 10 | Word Forms list + detail | DB services, Meanings (junction queries) | Reads junction table; needs meanings to exist |
| 11 | Categories view | Meanings (aggregation) | Aggregates over meanings |
| 12 | Timeline view | Meanings (date aggregation) | Chart over existing data |
| 13 | Gestures feature | DB services | Independent; no dependency on meanings/word forms |
| 14 | Doctor Report | All data layers, i18n (report namespace) | Reads across all entities |
| 15 | Settings: export/import | DB services (export/import service) | Operates on full DB snapshot |
| 16 | PWA manifest + service worker config | Stable app shell | Can only be verified end-to-end once build is stable |

## Anti-Patterns to Avoid

### Anti-Pattern 1: Importing `db` directly inside page or feature components

**What:** `import { db } from '../../../db/db'` inside a React component.
**Why bad:** Couples UI directly to storage; no decoupling seam for testing; triggers synchronous Dexie operations outside `useLiveQuery` observation context.
**Instead:** Call service functions for writes; use feature hooks (which call `useLiveQuery`) for reads.

### Anti-Pattern 2: Using `BrowserRouter` or `createBrowserRouter`

**What:** Standard history-API routing.
**Why bad:** GitHub Pages returns 404 for any path other than the root or explicit files. No `.htaccess` available. The 404-redirect workaround (custom 404.html that re-encodes the path) is fragile.
**Instead:** `createHashRouter` — URLs become `/#/dashboard`, which GitHub Pages serves correctly because the server only sees `index.html`.

### Anti-Pattern 3: Eager i18n loading with HTTP backend

**What:** Using `i18next-http-backend` to fetch `/locales/en/translation.json` at runtime.
**Why bad:** On first load offline, the network request fails before the service worker has cached the file. The app renders with missing translations.
**Instead:** Bundle locale files as static imports in `src/i18n/index.ts`.

### Anti-Pattern 4: Writing migration logic inside `version(1)` block

**What:** Adding an `upgrade()` call to the first version definition.
**Why bad:** Version 1 is only run on a fresh database; `upgrade()` is never called for v1 installs. Migrations belong on version 2+.
**Instead:** Start with a clean `version(1).stores(schema)`. Add `version(2).stores(additions).upgrade(fn)` when schema changes are needed.

### Anti-Pattern 5: Storing categories as a separate Dexie table with a junction

**What:** Creating a `categories` table and a `meaningCategories` junction for the fixed default category list.
**Why bad:** Categories are a closed, compile-time-known enum. A DB table adds schema complexity, migration surface, and query joins for zero runtime benefit.
**Instead:** Store categories as a `string[]` array field on the `Meaning` record with a multi-entry index (`*categories`) for efficient `where('categories').equals('food')` queries. Define the valid set as a TypeScript const.

## Scalability Considerations

This app is intentionally single-device and single-child. These limits are design decisions, not bugs.

| Concern | At MVP scale | If v2 adds multi-child |
|---------|-------------|----------------------|
| IndexedDB size | Negligible (<1MB for years of data) | Add `childId` FK to all tables; new index on every entity |
| Junction query performance | Instant; hundreds of records max | Still fast at thousands; IndexedDB compound indexes handle 100k+ rows |
| Service worker cache size | ~2MB for full app shell | Unchanged; SW only caches assets, not DB data |
| i18n bundle size | ~10KB for both locales | Add locale files per language; lazy loading becomes worthwhile at 5+ languages |
